URI #

Uniform Resource Identifier, 즉 리소스를 식별하는 통합된 방법이다.

URI와 URL, URN은 무엇이 다른가? 먼저 세 그룹의 포함 관계를 살펴보면 아래와 같다.

$$ {\text{URL}, \text{URN}} \subset \text{URI} $$

  • URI는 로케이터, 이름 또는 둘 다 추가로 분류될 수 있다.
  • URL(Resource Location); 위치를 지정한다.
  • URN(Resource Name); 이름을 지정한다.

아래는 URL의 예시이다.

foo://example.com:8042/over/there?name-ferret#nose

아래는 URN의 예시이다.

urn:example:animal:ferret:nose

URN은 현업에서 거의 사용되지 않고 거의 대부분이 URL을 사용한다.

URL, URN, URI에서의 Resource는 식별할 수 있는 모든 것을 의미한다.

URL #

scheme://[userinfo@]host[:port][/path][?query][#fragment]

e.g. https://www.google.com/search?q=hello&hl=ko

schema 프로토콜을 사용한다.

  • protocol: 어떤 방식으로 자원에 접근할 것인가를 정한 약속 규칙
    • e.g. http, https, ftp …
  • http는 80포트 https 주로 443 포트 사용

userinfo는 거의 사용하지 않음

host는 도메인 명을 거의 사용

  • port는 거의 생략 가능하고 http는 80, https는 443이 자동으로 들어감
  • path는 리소스가 있는 경로이며 계층적 구조로 되어 있따.
    • e.g. /home/file1.jpg, /members, /members/100, /item/iphone16

query는 key=value형태로 이루어짐

  • ?로 시작하며 &로 추가가 가능하다
  • e.g. ?keyA=valueA&keyB=valueB
  • query parameter, query string 등으로 불림
  • 웹서버에 제공하는 파라미터이며 문자 형태로 다 전송되므로

fragment는 잘 사용되지는 않음

  • html 내부 북마크 등에서 사용됨
  • 서버로 전송되는 정보는 아님

웹 브라우저 요청 흐름 #

https://www.google.com/search?q=hello&hl=ko

위의 링크로 접속하게 된다면 웹 브라우저가 아래와 같이 자동으로 변경시킨다.

https://www.google.com:443/search?q=hello&hl=ko

여기서 DNS 서버 조회를 통해 구글 서버의 IP 주소를 조회한다.

이후 HTTP 요청 메세지를 아래와 같이 보낸다.

GET /search?q=hello&hl=ko HTTP/1.1
Host: www.google.com

해당 HTTP Request Message를 아래와 같은 순서로 생성 및 전송한다.

  1. 웹 브라우저가 HTTP 메세지 생성
  2. SOCKET 라이브러리를 통해 전달
  • TCP/IP 연결(IP, PORT)
  • 데이터 전달
  1. TCP/IP 패킷 생성, HTTP 메시지 포함
